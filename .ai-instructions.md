# AIgentOS Coding Instructions

## Architecture

AIgentOS is a kernel-first framework for self-hosted AI agents. It runs as three Docker services:
- **kernel** — FastAPI app serving the HTTP API (`kernel/api/main.py`)
- **dialogue-worker** — background process that polls for pending user messages and streams LLM responses (`kernel/workers/dialogue_worker.py`)
- **webui** — optional React/Vite frontend (`agent-webui/`)

All three share a single SQLite database at `models-local/chat.db` (WAL mode).

## Key Conventions

1. **Shared utilities** live in `kernel/shared/`. Both `kernel/api/` and `kernel/workers/` import from here. Do not duplicate logic between them.
2. **Storage** is handled by `ChatStore` in `kernel/api/storage.py`. It uses a single reused SQLite connection per instance. All queries use parameterized inputs.
3. **LLM integration** goes through `OllamaClient` and `OllamaEmbeddingClient` in `kernel/api/llm.py`. Both reuse a persistent `httpx.AsyncClient`.
4. **Prompt system** uses YAML config (`agent-prompts/basic/agent.yaml`) with composable Markdown components. Prompt overrides are stored per-profile in the database.
5. **Token estimation** uses a `char_count / 4` heuristic. This is a known approximation documented with a TODO for future improvement via Ollama's tokenize endpoint.

## Module Boundaries

- `kernel/api/` — HTTP endpoints, Pydantic models, storage, LLM clients, prompt loading
- `kernel/workers/` — background processing (dialogue worker)
- `kernel/shared/` — pure utility functions shared across both (text processing, token metrics)
- `agent-prompts/` — prompt templates and components (Apache-2.0)
- `agent-webui/` — React frontend (Apache-2.0)

## Chat Flow

1. `POST /api/chat` creates a `pending` user event in `interaction_events` and returns `202 Accepted` with `conversation_id` and `event_id`.
2. The dialogue worker polls for pending user events, claims one via `BEGIN IMMEDIATE`, and sets it to `processing`.
3. The worker builds the LLM prompt: system prompt + retrieved memory chunks + conversation history + user message, applies context window pruning if near token limit.
4. The worker streams the LLM response via `OllamaClient`, updating the assistant event content on each chunk (enabling real-time SSE to the WebUI).
5. On completion, the worker stores embedding chunks for both user and assistant messages (if memory is enabled), runs memory compaction if needed, and records performance metrics.
6. The WebUI polls `GET /api/conversations/{id}/events` or connects via `GET /api/conversations/{id}/stream` (SSE) to get real-time updates.

## Memory System

- User and assistant messages are chunked (500 chars max) and embedded via `OllamaEmbeddingClient`.
- On each new message, the top-5 most similar prior chunks are retrieved via cosine similarity and injected as system context.
- When chunk count exceeds `MEMORY_CHUNK_LIMIT` (default 160), oldest chunks are summarized via LLM and re-embedded, then originals are deleted.
- Memory can be toggled via `memory_enabled` in context settings.
- All chunks are stored in `rag_chunks` with JSON-serialized embedding vectors.

## Database Schema

Key tables: `conversations`, `interaction_events`, `rag_chunks`, `performance_exchanges`, `prompt_profiles`, `prompt_component_overrides`, `prompt_context_settings`, `worker_heartbeat`.

- `interaction_events` and `performance_exchanges` cascade-delete from `conversations`.
- `rag_chunks` are manually cleaned when conversations are deleted (not FK-linked).
- `conversations` has denormalized `message_count` and `last_message_preview` columns maintained on event creation.
