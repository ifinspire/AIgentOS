# AIgentOS Coding Instructions

## Architecture

AIgentOS is a kernel-first framework for self-hosted AI agents. It runs as five Docker services:
- **kernel** — FastAPI app serving the HTTP API (`kernel/api/main.py`)
- **dialogue-worker** — background process that polls for pending user events, retrieves its own RAG context, streams LLM responses (`kernel/workers/dialogue_worker.py`)
- **orchestrator-worker** — background process that routes turns to tools (native or MCP) via an LLM call, handles post-turn finalization (memory writing, compaction), and document import via the MarkItDown MCP sidecar (`kernel/workers/orchestrator_worker.py`)
- **markitdown-mcp** — sidecar MCP server for document conversion (PDF, Office, etc. → Markdown). Runs the `markitdown-mcp` package over streamable HTTP.
- **webui** — optional React/Vite frontend (`agent-webui/`)

All kernel-side services share a single SQLite database at `models-local/chat.db` (WAL mode).

## Key Conventions

1. **Shared utilities** live in `kernel/shared/`. Both `kernel/api/` and `kernel/workers/` import from here. Do not duplicate logic between them.
2. **Storage** is handled by `ChatStore` in `kernel/api/storage.py`. It uses a single reused SQLite connection per instance. All queries use parameterized inputs.
3. **LLM integration** goes through `OllamaClient` and `OllamaEmbeddingClient` in `kernel/api/llm.py`. Both reuse a persistent `httpx.AsyncClient`.
4. **MCP integration** is in `kernel/api/mcp.py`. Supports `stdio` and `streamable_http` transports. Both do the full MCP initialize handshake per spec (2025-03-26). Shared bootstrap functions (`ensure_default_markitdown_server`, `discover_enabled_servers`) are used by both the kernel API and orchestrator worker.
5. **Prompt system** uses YAML config (`agent-prompts/basic/agent.yaml`) with composable Markdown components. Prompt overrides are stored per-profile in the database.
6. **Token estimation** uses a `char_count / 4` heuristic. This is a known approximation documented with a TODO for future improvement via Ollama's tokenize endpoint.

## Module Boundaries

- `kernel/api/` — HTTP endpoints, Pydantic models, storage, LLM clients, MCP client, prompt loading
- `kernel/api/mcp.py` — MCP protocol client (stdio + HTTP transports), tool discovery, tool calling, shared bootstrap
- `kernel/workers/` — background processing (dialogue worker, orchestrator worker)
- `kernel/shared/` — pure utility functions shared across both (text processing, token metrics)
- `agent-prompts/basic/` — dialogue model prompt components (Apache-2.0)
- `agent-prompts/orchestrator/` — orchestrator routing prompt (`routing.md`) and math subagent prompt (`math_subagent.md`); edit to add/remove/reconfigure native tools without touching Python code
- `agent-webui/` — React frontend (Apache-2.0)

## Chat Flow

1. `POST /api/chat` creates a `pending` user event in `interaction_events` and a `prepare_turn` orchestration event, then returns `202 Accepted` with `conversation_id` and `event_id`.
2. The **dialogue worker** and **orchestrator worker** claim their respective events independently and run in parallel:
   - **Dialogue worker**: embeds the user message and retrieves the top-5 most similar memory chunks via cosine similarity (owns its own RAG). Then waits up to 1.5 s for the orchestrator's tool-routing signal.
   - **Orchestrator worker**: builds the routing prompt by combining `routing.md` with dynamically discovered MCP tools (names, descriptions, parameter schemas). Makes an LLM routing call to decide if a native tool, MCP tool, or no tool should run. If a tool is selected, executes it and writes the result to `turn_contexts.tool_observations`. Posts a `role: "tool"` interaction event for auditability.
3. The **dialogue worker** reads `tool_observations` from `turn_contexts`, then builds the LLM prompt: system prompt + retrieved chunks + tool observations + document context + conversation history + user message. Applies context window pruning if near token limit.
4. The dialogue worker streams the LLM response via `OllamaClient`, updating the assistant event content on each chunk (enabling real-time SSE to the WebUI). If a tool observation matches a deterministic pattern (calculator, letter counter, math subagent), the response is emitted directly without calling the LLM.
5. On completion, the dialogue worker records performance metrics and queues a `finalize_turn` orchestration event.
6. The **orchestrator worker** claims `finalize_turn`, stores embedding chunks for both user and assistant messages, runs memory compaction if needed, and selects memory candidates via LLM.
7. The WebUI polls `GET /api/conversations/{id}/events` or connects via `GET /api/conversations/{id}/stream` (SSE) to get real-time updates.

## MCP System

- MCP servers are registered in the `mcp_servers` table with transport config (stdio command+args or HTTP URL), enabled state, and discovered tools.
- On startup, both the kernel API and orchestrator worker call `ensure_default_markitdown_server` (registers the bundled MarkItDown MCP if missing) and `discover_enabled_servers` (runs `tools/list` on all enabled servers with retry logic).
- Discovered tools store full metadata: `name`, `description`, and `inputSchema`. This metadata is used to build richer routing prompts.
- MCP tool identifiers use slugified server names: `mcp::<server-slug>::<tool-name>` (e.g. `mcp::markitdown-mcp::convert_to_markdown`).
- The MCP dashboard in the WebUI allows registering, enabling/disabling, refreshing, and deleting servers.
- Document import routes through the bundled MarkItDown MCP sidecar rather than in-process `markitdown` library.
- Both stdio and HTTP transports create a new connection per call. This is intentional for the single-user self-host target — persistent connections would add complexity for little gain at this scale.

## Memory System

- User and assistant messages are chunked (500 chars max) and embedded via `OllamaEmbeddingClient`.
- On each new message, the top-5 most similar prior chunks are retrieved via cosine similarity and injected as system context.
- When chunk count exceeds `MEMORY_CHUNK_LIMIT` (default 160), oldest chunks are summarized via LLM and re-embedded, then originals are deleted.
- Memory can be toggled via `memory_enabled` in context settings.
- All chunks are stored in `rag_chunks` with JSON-serialized embedding vectors.

## Tool Registry

Native tools are declared in `agent-prompts/orchestrator/routing.md`. The orchestrator loads this file at runtime, appends discovered MCP tools with their descriptions and parameter schemas, and passes the combined prompt to the routing LLM. To add a new native tool:
1. Add a description entry in `routing.md`.
2. Add a handler branch in `_process_prepare_turn` in `orchestrator_worker.py`.
3. Add a `_direct_tool_response` case in `dialogue_worker.py` for deterministic formatting (optional — without this, the LLM generates the response using the tool observation as context).

To add an MCP tool: register the MCP server via the dashboard or `POST /api/mcp/servers`. Discovered tools are automatically added to the routing prompt.

Available native tools: `math_subagent` (LLM-assisted arithmetic with context carry-forward), `calculate` (AST-safe arithmetic), `count_occurrences` (character frequency). All execute in the orchestrator synchronously within the `prepare_turn` event.

## Database Schema

Key tables: `conversations`, `interaction_events`, `rag_chunks`, `performance_exchanges`, `prompt_profiles`, `prompt_component_overrides`, `prompt_context_settings`, `turn_contexts`, `orchestration_events`, `document_imports`, `mcp_servers`, `worker_heartbeat`.

- `interaction_events` and `performance_exchanges` cascade-delete from `conversations`.
- `rag_chunks` are manually cleaned when conversations are deleted (not FK-linked).
- `conversations` has denormalized `message_count` and `last_message_preview` columns maintained on event creation.
- `turn_contexts.retrieved_chunks` is always written as `[]` by the orchestrator worker (RAG is owned by the dialogue worker and is not persisted to `turn_contexts`). The column is retained for schema compatibility.
- `interaction_events` with `role = "tool"` are written by the orchestrator after tool execution. They are excluded from the dialogue LLM's conversation history (only `user` and `assistant` roles are included) but appear in the event stream for UI auditability.
- `mcp_servers` stores registered MCP servers with `discovered_tools` as a JSON column containing tool metadata (name, description, inputSchema).

## Testing

- Kernel tests run in Docker, not from the host Python environment. Use `docker compose run --rm kernel-test` for the full suite.
- For the CI-equivalent fast lane, run `docker compose run --rm kernel-test sh -c "mkdir -p test-results && pytest -v tests -m bvt --junitxml=test-results/unit-pytest.xml"`.
- GitHub Actions CI runs only the `bvt` subset for the kernel job and publishes the JUnit XML report from `test-results/unit-pytest.xml`.
- Pytest markers are registered in `pytest.ini`:
  - `bvt` = Build Verification Tests. Fast happy-path checks that should stay green on every CI run.
  - `p0` = Highest-priority regression coverage. Release-blocking tests; `bvt` is a subset of `p0`.
- When adding or updating tests, prefer explicit per-test decorators (`@pytest.mark.p0`, `@pytest.mark.bvt`) so the classification is visible at the test definition.
